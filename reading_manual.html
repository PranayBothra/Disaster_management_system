AI Disaster Response System

Technical Handover Document detailing the backend architecture, data flow, and pipeline execution workflow.

01 Executive Summary

The AI Disaster Response System is a sequential algorithmic pipeline built in Python. The backend is designed to simulate a rescue operation scenario by orchestrating four distinct artificial intelligence methodologies: topological mapping (BFS), geometric routing (A*), probabilistic reasoning (Bayesian Inference), and stochastic combinatorial optimization (Simulated Annealing).

The entire execution lifecycle is centralized and managed via a single orchestrator function within the main application layer, ensuring a strictly synchronous and deterministic data flow from raw map generation to final route visualization.

02 Backend Architecture & Data Flow

The backend operates on a unidirectional data flow. Data state is passed as serialized dictionaries through the four primary modules. The orchestrator maintains the session state and passes outputs from one module as inputs to the next.

03 Detailed Pipeline Workflow

The backend logic is strictly governed by the run_pipeline() function in main.py. The workflow executes in the following exact sequence:

Phase 1: Environment Initialization & Reconnaissance

  1. Graph Construction: group1_map.graph.build_graph() is invoked to generate the node dictionary and adjacency list. Nodes are categorized by string prefixes (e.g., "G" for Gate, "B" for Building).
  2. Reachability Analysis: group1_map.bfs.get_reachable_nodes(graph, start="G") executes a Breadth-First Search originating from the Gate. It returns a set of reachable node IDs.
  3. Target Selection: The orchestrator filters the reachable nodes. Only nodes beginning with "B" (Buildings) are added to the target_nodes list.

Phase 2: Pairwise Routing & Cost Matrix Generation

  1. Permutation Setup: A list of points_of_interest is created, combining the start node ("G") and all discovered target_nodes.
  2. Matrix Calculation: The system iterates through every unique pair (u, v) in the points of interest.
  3. A* Execution: For each pair, group2_routing.astar.a_star(graph, nodes, u, v) calculates the optimal path using Euclidean distance as the heuristic.
  4. Caching: The backend populates two global-level dictionaries: path_lookup[(u, v)] stores the ordered list of node IDs forming the route, and distance_lookup[(u, v)] stores the cumulative path cost.

Phase 3: Sensor Simulation & Probabilistic Inference

  1. Evidence Generation: For each target building, the orchestrator simulates incoming data by assigning a random risk_level ("Low", "Medium", "High") and a random sensor_reading ("None", "Faint", "Detected").
  2. Bayesian Update: group3_bayesian.bayesian.update_survivor_probability() is called with the generated evidence.
  3. Posterior Calculation: The module performs sum-rule marginalization against hardcoded Conditional Probability Tables (CPTs) to output a float representing the final probability of human presence.

Phase 4: Simulated Annealing Optimization

  1. Score Assignment: group4_planning.assignment.calculate_priority_score() evaluates each target, combining the Bayesian probability with static vulnerability modifiers to assign a numerical weight.
  2. State Initialization: An initial route sequence is formulated (Gate -> T1 -> T2 ... -> Gate).
  3. Annealing Loop: group4_planning.optimization.simulated_annealing() begins. It proposes neighbor states by randomly swapping two targets in the sequence.
  4. Cost Function: The fitness of a sequence is evaluated by summing the distances from distance_lookup and subtracting the priority weights of visited nodes (encouraging high-priority targets earlier in the route).
  5. Cooling: The loop accepts better routes unconditionally and worse routes probabilistically based on the current "temperature", which decays over a set number of iterations.

04 Core Data Structures & State Management

The backend utilizes native Python dictionaries heavily for state transfer across module boundaries. Understanding these dictionary schemas is critical for maintaining the backend.

1. The Node Dictionary Schema

Generated by Group 1 and passed to all subsequent groups. Keys are node string identifiers (e.g., 'G', 'B1').

nodes = {
    "G": {
        "coord": (0.0, 0.0), 
        "kind": "Gate"
    },
    "B1": {
        "coord": (1.5, 2.0), 
        "kind": "Building"
    }
}

2. The Graph (Adjacency List)

Represents the topological edges. Keys are origin nodes, values are lists of connected destination nodes.

graph = {
    "G": ["R1", "R2"],
    "R1": ["G", "B1", "I1"]
}

3. Survivor State Dictionary

Maintained by the orchestrator after Group 3 execution to hold dynamic target data.

survivor_data = {
    "B1": {
        "id": "B1",
        "risk_level": "High",
        "sensor_reading": "Detected",
        "prob": 0.85,  # Calculated by Group 3
        "vulnerability": "Elderly"
    }
}

4. A* Cache Lookups

Flat dictionaries using tuple keys representing (Start, End) node pairs. This ensures O(1) retrieval during the thousands of iterations in Group 4's simulated annealing.

# distance_lookup schema
distance_lookup = {
    ("G", "B1"): 3.54,
    ("B1", "G"): 3.54
}

# path_lookup schema
path_lookup = {
    ("G", "B1"): ["G", "R1", "B1"]
}

05 Backend Module Responsibilities

Group 1: Map Generation (group1_map/)

  • graph.py: Contains hardcoded spatial configurations and the build_graph logic. Responsible for translating layout definitions into the standardized Node and Graph dictionary structures.
  • bfs.py: Implements standard queue-based Breadth-First Search to return a set of reachable keys.

Group 2: Routing (group2_routing/)

  • heuristic.py: Extracts "coord" tuples from the node dictionary and applies standard math.hypot for Euclidean calculations.
  • astar.py: Standard implementation using Python's heapq. It is completely stateless and relies entirely on inputs provided by the orchestrator per pair calculation.

Group 3: Inference (group3_bayesian/)

  • cpt.py: Houses the static Conditional Probability Tables formatted as nested dictionaries.
  • bayesian.py: Performs pure mathematical marginalization. It takes strings ("High", "Detected") and retrieves the associated float from the CPTs to calculate joint and marginal probabilities.

Group 4: Optimization (group4_planning/)

  • assignment.py: Contains the static heuristic mapping that converts survivor vulnerability data into scalar weight values.
  • optimization.py: Houses the simulated_annealing loop. It requires the pre-computed distance_matrix to evaluate route permutations without triggering pathfinding during the annealing loop.

06 Setup & Deployment

Environment Setup

  • Python 3.9+
  • Standard virtual environment isolation

Execution Commands

# 1. Clone the repository and navigate to the directory
cd disaster_ai

# 2. Create and activate a virtual environment
python -m venv venv
source venv/bin/activate  # On Windows: venv\Scripts\activate

# 3. Install dependencies
pip install -r requirements.txt

# 4. Initialize the backend orchestrator via Streamlit
streamlit run main.py

07 Known Logs & Behavior

Deployment Port Binding

The backend relies on the Streamlit server for execution. Note the following historical log behavior regarding server initialization:

2026-04-26 01:43:04.974 Port 8503 is not available

Behavior Note: By design, if the default port (8501) or subsequent ports are bound by existing background processes, the server will incrementally search for the next available port. A manual port override can be passed to the execution command if a specific deployment port is required: streamlit run main.py --server.port 8080.
